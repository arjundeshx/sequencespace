Memory Efficient Calculation of RMSD and RMSF with MDTraj and MDAnalysis -
A Tutorial

A basic simulation post-processing walkthrough to help you handle large trajectories using MDTraj and MDAnalysis, two widely used frameworks for analyzing molecular dynamics simulations.

DateMar 14, 2026
Read time10 min
TopicStructural Biology
CodeGitHub ↗

When analyzing large molecular dynamics (MD) trajectories, it can be difficult to make sense of the data. The easiest and simplest ways to get started analyzing and comparing the movement of proteins is by generating plots for root mean square deviation (RMSD) and per-residue root mean sequare fluctuation (RMSF).

In my experience, the MDTraj an MDAnalysis python packages have been immensely useful for the analysis of molecular dynamics (MD) simulations; they are highly programmable and cutomizeable. Despite this, it can be cumbersome to parse through the documentation for these libraries, and dealing with large trajectories is often memory intensive. In this tutorial, I will cover the basic methods I use to calculate per-residue RMSF and backbone RMSD in a memory-safe way that works on any personal computer.

Calculating RMSD with MDTraj

RMSD is essentially a measure of the distance from a conformation of a biomolecule (e.g a protein) to some reference structure. RMSD calculations work by measuring the 3-dimensional distance between reference atoms and corresponding atoms in a particular frame of a trajectory, squaring and taking the average of these squared distances across all of the atom pairs being compared, and finally taking the square root. The function of squaring is to eliminate negative distances so as to capture movements in opposing directions as an RMSD of > 0 (i.e if one part of an object moved right, while the another part moved equivalently left from a reference, we would not want those to cancel out - eliminating signs is the function of squaring in RMSD).

"The structure is the vocabulary. The dynamics are the language."

This post will deal with the RMSD over time of proteins. A change in RMSD indicates a change in the conformation of a protein, and different conformations show up as stable, flat-lines on RMSD vs. time graphs. For our purposes, we compare each frame of a trajectory to the starting structure (frame = 0) as a reference to measure divergence from the starting structure over time. Thus, looking at RMSD vs. time plots will enable us to see long-term conformational changes.

📐 Formula
Root Mean Square Deviation (RMSD)
\[ \text{RMSD}(t) = \sqrt{\frac{1}{N} \sum_{i=1}^{N} \left\| r_i(t) - r_i^{\text{ref}} \right\|^2} \]
where \(N\) is the number of atoms, \(r_i(t)\) is the position of atom \(i\) at time \(t\), and \(r_i^{\text{ref}}\) is its position in the reference structure.

For our purposes we will restrict the atoms being compared in the RMSD calculation to the alpha carbons of the protein backbone - amino acid side chains are flexible and move around a lot over the course of a typical MD simulation, and when looking for a change in shape within a protein what we really care about is a change in the structure of the BACKBONE of a protein. This is why it can be convenient to use solely alpha carbons for RMSD calculations.

The Memory Problem

By default, MDTraj loads in the entire trajectory when using the function md.load() (which will be demonstrated in the code that follows) - this can be a huge problem for large trajectories. Attempting to load in an entire trajectory into RAM fills it up quickly - this will break your analysis code, or make it impossibly slow. Fortunately, MDTraj includes the function md.iterload() which allows us to load in chunks of a trajectory at a time for comparison and RMSD calculation without filling up memory.

The Code

The open-weight AF3 release requires some setup. Here's the pipeline I use for batch predictions on a protein family of interest: 

import mdtraj as md
      import numpy as np
      import matplotlib.pyplot as plt
      from matplotlib.backends.backend_pdf import PdfPages
      
      WT_TRAJ  = "/path/to/wildtype_production.dcd"
      MUT_TRAJ = "/path/to/mutant_production.dcd"
      
      WT_TOP   = "/path/to/wildtype_topology.psf"
      MUT_TOP  = "/path/to/mutant_topology.psf"
      
def compute_rmsd(traj_file, top_file, frame_interval_ps=50.0, chunk_size=10):

        top    = md.load_topology(top_file)
        ca     = top.select("protein and name CA")
        ref    = md.load(traj_file, top=top_file, frame=0)
        ref_ca = ref.atom_slice(ca)
    
        rmsd_values   = []
        time_values   = []
        frame_counter = 0
    
        for chunk in md.iterload(traj_file, top=top_file, chunk=chunk_size):
            chunk = chunk.atom_slice(ca)
            chunk.superpose(ref_ca)
    
            rmsd_chunk = md.rmsd(chunk, ref_ca)
    
            n       = chunk.n_frames
            time_ps = np.arange(frame_counter, frame_counter + n) * frame_interval_ps
            time_ns = time_ps / 1000.0
    
            rmsd_values.append(rmsd_chunk)
            time_values.append(time_ns)
            frame_counter += n
    
        return np.concatenate(time_values), np.concatenate(rmsd_values)
    
time_wt,  rmsd_wt  = compute_rmsd(WT_TRAJ,  WT_TOP)
      time_mut, rmsd_mut = compute_rmsd(MUT_TRAJ, MUT_TOP)
      
fig, ax = plt.subplots(figsize=(8, 6))

        ax.plot(time_wt,  rmsd_wt,  color="steelblue",   linewidth=1, label="Wild-Type")
        ax.plot(time_mut, rmsd_mut, color="lightskyblue", linewidth=1, label="Mutant")
        
        ax.set_xlabel("Time (ns)")
        ax.set_ylabel("RMSD (nm)")
        ax.set_title("Protein Cα RMSD vs Time — Wild-Type vs Mutant")
        ax.legend()
        ax.grid(True, alpha=0.3)
        
        with PdfPages("RMSD_WT_vs_Mutant.pdf") as pdf:
            pdf.savefig(fig)
        
        plt.close(fig)

What Comes After Structure?

The most exciting frontier, in my view, is not better static structure prediction — it's using predicted structures as priors for dynamic simulation. Tools like OpenMM and GROMACS can now be seeded with AF3 coordinates to run molecular dynamics, and the combination is genuinely powerful.

There's also a growing ecosystem of structure-aware machine learning models — GVP, EquiFold, and others — that take structural representations as input and predict function, binding affinity, or mutational effects. The bottleneck is no longer getting structures; it's knowing what to do with them.

My Current Workflow

For a recent project predicting the effect of point mutations on binding stability, my pipeline looked like this: AF3 prediction → ensemble generation with 5 random seeds → per-residue confidence filtering → Rosetta ddG calculation → ESM-1v zero-shot mutant scoring → ensemble averaging. The combination outperformed any single method.

Conclusion

AlphaFold 3 is a genuinely transformative tool. But transformative tools don't end fields — they open new ones. The protein folding problem, narrowly defined, is largely solved. The problems of protein dynamics, disorder, and function remain wide open.

If you're a computationalist just getting into structural biology, this is an exciting time. The infrastructure is finally good enough that you can ask biological questions rather than engineering questions. Don't waste that by treating AF3 as an oracle.
